[Django] Authentication 인증부를 구현하기 위해 알아야 할 것들

Oct 13, 2021 in Django, Authenticate

Authentication은 요청을 보낼 때 인증된 사용자나 Token 인지를 식별하는 역할을 한다. DRF에서 제공하고있는 인증 클래스를 이용하면 인증부를 구현할 수 있는데, 가장 기본적인 클래스는 BaseAuthentication 이 있다.

BaseAuthentication은 사용자의 username과 password를 기반으로 한 인증이기 때문에 가장 기본적이며, 보통 테스트 환경에서 적합한 인증 방법이다. 상용 환경에서 BaseAuthentication를 단독으로 사용하려면 API를 https를 통해서만 사용할 수 있는지, 항상 username/password 를 함께 요청할 수 있는지에 대한 고려가 필요하다.

따라서 보통은, 간단하게 Token 기반의 인증을 할 수 있는 TokenAuthentication을 사용한다.
TokenAuthentication 은 BaseAuthentication을 상속받아 구현이 되어있으며, 이번에는 TokenAuthentication을 이용하여 인증부를 구현할때 알아야 할 것들에 대해서 작성해본다.

기본적인 Authentication 클래스 DRF 설정에 관련한 설명은 아래 DRF 공식문서로 대체한다.

1. Token Model

TokenAuthentication 인증 클래스를 사용하려면 토큰 정보를 저장할 테이블이 필요하다. Token Model은 직접 구성하여 사용할 수도 있고, DRF의 rest_framework.authtoken APP을 추가하여 사용할 수도 있다. (django migration을 제공한다.)

Token 의 주된 필드는 key 와 user 이다.

  • key : Token 의 PK 이며, 인증에 사용되는 토큰 키다.
  • user : 해당 key 를 발급 받을 user를 말한다.
    • settings 파일에서 AUTH_USER_MODEL 에 User Model 을 지정하여 사용한다.

2. .authenticate() 알아보기

Authentication 인증 클래스의 핵심 메서드이다.
만약 인증 로직을 커스텀 하고 싶다면 .authenticate() 를 Override 하여 사용한다. 해당 메서드는 커스텀이 되더라도 아래 몇가지 기본적인 틀을 갖추고 있어야한다.

  • 인증이 성공하면 (user, auth) 형태의 Tuple을 return 해야한다.

  • 인증에 시도했지만 실패하는 경우에는 AuthenticationFailed Exception 을 raise 하거나, None 을 return 해야한다.

  • 인증이 아예 시도되지 않은 경우라면, None을 반환한다.

    • 사용중인 다른 Authentication 인증 Class 로직을 탈 수 있어야 하기 때문이다. (Authentication은 다수개 지정이 가능하다.)

    3번째 항목에서 알 수 있겠지만, .authenticate() 에서는 인증을 시도하지 않은 경우에 대한 예외처리가 되지 않는다. 이는 인증(authenticate)의 영역이 아니라, 권한(permission)의 영역이기 때문이다. 이에 대한 예외처리를 하기 위해서는 rest_framework에서 제공하는 IsAuthenticated Permission 을 사용한다.

    IsAuthenticated Permission 은 인증된 유저가 존재하지 않을 경우 Exception을 낸다. Permission 이라 403 Forbidden 응답을 낼 것 같지만, rest_framework 내부에서 인증되지 않는 경우를 분기처리하여 401 UnAuthorized 응답을 내린다.

.authenticate() 에서는 공백을 기준으로 keyword와 Token key 를 나눠서 값을 검증한다.

  • keyword : 클래스 변수로 지정된 keword 값과 일치하는지 검증
    • e.g. Token
  • Token key : Token 테이블을 조회하여 해당 key 를 가지고 있는 User가 존재하는지 검증
    • e.g. 9944b09199c62bcf9418ad846dd0e4bbdfc6ee4b

검증에 통과한다면 위에서 설명한 것 처럼 (user, auth) 형태의 튜플을 return 한다.

  • user : AUTH_USER_MODEL 으로 지정했던 User 모델이 담긴다.
  • auth : 인증을 시도한 Key를 PK로 가진 Token 모델이 담긴다.

그리고 해당 정보들은 perform_authentication() 시점에 호출되는 _authenticate()에서 인증에 통과할 경우 실행되는 로직에 따라
request.user와 request.auth 로 담기기 때문에 view 에서는 언제든지 인증된 사용자 정보와 Token 에 접근할 수 있게 된다.
