[MySQL] Explain 실행계획

Sep 08, 2021 in MySQL

Slow Query를 없애는 것은 성능을 향상시키기 위한 매우 중요한 수단이다. 쿼리 성능 진단의 첫 걸음은 실행한 SQL이 DB에서 어떻게 처리되는 지를 파악하는 것인데, 그 수단이 바로 오늘 알아볼 MySQL Explain이다.

Explain ?

MySQL 옵티마이저는 어떤 실행 계획으로 쿼리를 실행 했을 때 가장 적은 비용이 드는지 계산한다. Explain 은 이러한 실행계획이 무엇인지 알고 싶을 때 사용하는 명령어이다. 실행 계획을 활용하면 기존의 쿼리를 튜닝할 수 있을 뿐만 아니라, 성능 분석, 인덱스 전략 수립 등과 같이 성능 최적화에 대한 업무를 처리할 수 있다.

  • 질의하고자 하는 select 쿼리 앞에 explain 키워드를 붙여 사용한다.

Explain 결과

id select_type table partitions type possible_keys key key_len ref rows filtered Extra
1 PRIMARY A NULL const PRIMARY PRIMARY 4 const 1 100 NULL
1 PRIMARY B NULL ref info_goods_index_fk, XIF1_goods_info info_goods_goods_index_fk 5 const 2 100 Using where
1 PRIMARY C NULL eq_ref PRIMARY PRIMARY 4 A.file_manage_index 1 100 NULL
3 DEPENDENT SUBQUERY U1 NULL ref XIF1_goods_info XIF1_goods_info 5 B.goods_index 2 10 Using where; Using filesort
2 DEPENDENT SUBQUERY U0 NULL ref info_goods_index_fk info_goods_index_fk 5 B.goods_index 5 10 Using where; Using filesort

각 컬럼별 의미를 알아보고 위의 결과를 해석해보자.

  • id

    • select 쿼리 별로 부여되는 식별자 값, 실행 순서
    • 여러개의 테이블을 조인하면, 조인되는 테이블의 개수만큼 실행계획 레코드가 출력되지만 같은 id가 부여된다.
    • 서브쿼리 등으로 구성되어 있으면 각 레코드에 다른 id가 부여된다.

  • select_type

    • select 문 유형
    • PRIMARY : 가장 바깥 select 문
    • DEPENDENT SUBQUERY : 서브쿼리에서 바깥쪽 select문에서 정의된 컬럼을 사용하는 경우, 이는 서브쿼리가 외부 쿼리 결과에 의존적이기 때문에 전체 쿼리의 성능을 느리게 만든다.
    그 외에 나올수 있는 select_type
    • SIMPLE : 단순 SELECT
    • UNION : UNION 쿼리에서 Primary를 제외한 나머지 SELECT
    • DEPENDENT_UNION : UNION 과 동일하나, 외부 쿼리에 의존적 (값을 공급 받음)
    • UNION_RESULT : UNION 쿼리의 결과물
    • SUBQUERY : Sub Query 또는 Sub Query를 구성하는 여러 쿼리 중 첫 번째 SELECT문
    • DERIVED : SELECT로 추출된 테이블 (FROM 절 에서의 서브쿼리 또는 Inline View)
    • UNCACHEABLE SUBQUERY : UNION 과 동일하지만 공급되는 모든 값에 대하여 UNION 쿼리를 재처리

  • type

    • 접근 방식! 이는 대상 테이블로의 접근 방식이 효율적일지 여부를 판단하는데 중요한 항목이 된다.
    • const : primary key나 unique key 를 이용하는 where절 조건을 가지고있으며, 반드시 1건만을 반환한다.
    • eq_ref : primary key나 unique key 를 사용하여 조인하는 경우이며, 조인되는 테이블에서 레코드가 반드시 한건만 존재한다는 보장이 있어야 사용할 수 있다.
    • ref : primary key나 unique key 가 아닌 key로 조인을 하는 경우이며, 반환되는 레코드가 반드시 한 건이라는 보장이 없으므로 eq_ref 보다는 느리다.
    그 외에 나올수 있는 type
    • system : 테이블에 1행밖에 없는 경우
    • ref_or_null : ref 와 같지만 null 이 추가되어 검색되는 경우
    • index_merge : 여러개의 인스턴스를 사용해 행을 가져오고 그 결과를 통합하는 경우
    • unique_subquery : IN 서브쿼리 접근에서 Primary key를 사용하는 경우
      • SELECT * FROM A WHERE A.pk IN (SELECT Primary Key FROM B);
    • index_subquery : unique_subquery와 비슷하나 Primary Key가 아닌 인덱스인 경우
      • SELECT * FROM A WHERE A.pk IN (SELECT B.key FROM B);
    • range : 인덱스 특정 범위의 행에 접근하는 경우
    • fulltext : fulltext 인덱스를 사용한 경우
    • index : 인덱스를 처음부터 끝까지 찾아서 검색하는 경우로, 일반적으로 인덱스 풀스캔이라고 함
    • all : 테이블을 처음부터 끝까지 검색하는 경우로, 일반적으로 테이블 풀스캔이라고 함

  • possible_key

    • 최적의 실행 계획을 만들기 위해 후보로 선정했던 인덱스들

  • key

    • possible_key 중에서 실제로 사용한 인덱스

  • rows

    • 접근 방식을 이용해, 쿼리 수행시 접근해야하는 데이터 행 수 (예측값임)

  • filtered

    • 가져온 레코드 중, where 조건 검색에 필터링되어 남은 레코드 개수 추정치 (단위 : %)

  • Extra

    • 옵티마이저가 동작하는데에 대해서 우리에게 알려주는 힌트
    • Using where : 레코드를 별도의 가공이나 연산 작업을 해서 필터링 한 경우
    • Using filesort : order by 처리시 인덱스를 활용하지 못했을 경우
    그 외에 나올수 있는 Extra
    • using index : 테이블에는 접근하지 않고 인덱스에서만 접근해서 쿼리를 해결하는 것을 의미 (커버링 인덱스)
    • using temporary : 쿼리 처리 시 내부적으로 temporary table이 사용되는 경우를 의미
    • Using index for group-by : Using index와 유사하지만 GROUP BY가 포함되어 있는 쿼리를 커버링 인덱스로 해결할 수 있음
    • 등등…

오늘의 결론!

[Clean Code] 코드를 클린하게 (1) : 이름, 함수

Jun 23, 2021 in 클린코드

클린코드 책을 읽고 정리하는 글입니다.

나쁜 코드를 작성하면 안 되는 이유

나쁜 코드는 개발 속도를 크게 떨어뜨린다. 코드를 고칠 때마다 엉뚱한 곳에서 문제가 생기기 때문이다. 기한을 맞추려면 나쁜 코드를 양산할 수 밖에 없다고 생각할 수 있다. 하지만 틀렸다. 오히려 엉망진창인 상태로 인해 속도가 곧바로 늦어지고, 결국 기한을 놓친다. 빨리 가는 유일한 방법은, 언제나 코드를 최대한 깨끗하게 유지하는 습관을 가지는 것이다.

어떤 코드를 깨끗한 코드라 하는 걸까?

깨끗한 코드는 누군가가 주의 깊게 짰다는 느낌을 준다. 작성자가 이미 모든 사항을 고려 했기 때문에 고치려고 살펴봐도 딱히 손 댈 곳이 없다. 사실 진짜 깨끗한 코드는 너무도 잘 짜놓은 코드라 읽는 이가 그 사실을 모르고 넘어간다.

읽기 쉬운 코드가 중요한 이유

코드는 읽는 시간 대 코드를 짜는 시간 비율이 10:1을 훌쩍 넘는다. 새코드를 짜면서도 우리는 끊임없이 기존코드를 읽기 때문이다. 따라서 읽기 쉽게 만들면 짜기도 쉬워진다.


그럼 이제 클린코드를 어떻게 작성하는 지에 대해 알아보자.

의미있는 이름

  1. 의도를 분명히 밝히면 함수가 하는 일을 이해하기 쉬워진다.
  2. 의미있게 구분해라. 단순히 연속적인 숫자를 덧붙이거나 아무 정보를 제공하지 못하는 이름은 피해야한다. 최악은 ‘이미 a,b 변수가 있으니까 c로 써야지~’ 하는 것이다.
  3. 검색하기 쉬운 이름을 사용하라. 상수로 예를 들면 쉽다. MAX_COUNT=20 일 경우, 20으로 검색하면 별에 별 검색결과가 다 나올 것이다.
  4. 한 개념에 한 단어를 사용하라. 데이터를 입력할 때 add, insert, create 등 다양하게 표현할 수 있지만 제각각 표현하면 혼란스럽다.

함수

  1. 작게 만들자. 들여쓰기 수준은 1단에서 2단 정도가 적당하다.
  2. 한 가지만 하라. 한 가지를 판단하는 방법이 어려울 수 있다. 그럴 땐 지정된 함수 이름 아래서 추상화 수준이 한 단계만 수행하는지를 확인하라.
  3. 서술적인 이름을 사용하라. 이름이 길어도 괜찮다. 길고 서술적인 이름이 서술적인 주석보다 좋다.
  4. 함수의 인수는 0개가 가장 이상적이고, 4개 이상은 특별한 이유가 있어야한다. 객체를 생성해서 인수를 줄이는 방법이 눈속임처럼 보일 수 있지만, 객체의 이름을 지어주기 때문에 결국은 개념을 표현하게 된다.
  5. 부수효과(Side effect)를 일으키지 마라. 예상하지 못한 부수적인 효과를 일으킬 바에는 아예 이름을 바꿔서 예측 가능하게 하자. 위 함수는 이름만봐서는 세션초기화를 예측하지 못한다. 사실 이 함수는 세션을 초기화해도 괜찮은 경우에만 호출이 가능한 함수다. 이럴 경우에는 checkPasswordAndInitializeSession 라는 이름이 훨씬 좋다.
  6. 오류코드보다 예외를 사용하자. 위 처럼 작성하지 말자. Try/Catch(예외)를 사용하면, 오류 처리 코드가 원래 코드에서 분리되므로 코드가 깔끔해진다.

오늘의 결론!

아는 것과 실천하는 것은 다르다.

[Django] Unit Test의 Mocking! @patch

May 12, 2021 in Django

Mocking ?

Mock 은 ‘거짓된, 가짜의’ 라는 사전적 의미를 가진 것 처럼, 테스트코드를 작성할 때 가짜 값을 사용할 수 있게 하는 객체이다. 이 Mock 객체를 이용하여, 단위 테스트 시에 외부에 의존하는 부분을 임의의 가짜로 대체하여 테스트 하는 상황이 자주 발생하는데, 이것을 ‘Mocking’ 이라고 한다.

Mocking 방법은 2가지가 있다.

  1. 직접 Mock 객체를 생성하여 사용
  2. unittest.mock 모듈의 @patch 데코레이터를 사용

이 글은 Mocking시 주로 사용되는 Mock객체의 속성에 대해 먼저 알아보고, @patch 데코레이터를 사용하는 방법에 대해 작성한다.

1. Mock

@patch 에서 사용되는 Mock 객체의 주요 속성을 알아보자.

  • return_value: 반환되는 값을 설정한다.

  • side_effect : 설정하는 값에 따라 동작이 다르다.

    • Exception 클래스를 지정하면 에러를 발생한다.
    • 함수를 지정하면 해당 함수를 호출한다.
    • Iterable(ex.list)을 지정하면 mock 호출 순서대로 해당 값을 반환한다.

2. @patch

unittest.mock 모듈의 @patch 데코레이터를 이용하면, 지정한 특정 메서드를 Mocking하여 해당 메서드가 호출될 때 mock 객체로 patch 한다. patch된 mock 객체는 테스트가 끝나면 원래대로 복원된다.

  1. @patch('example_api.client.ExampleAPI.get')
    Mocking 이 필요한 메서드를 지정하는 부분이다.
    지정할 메서드의 path는 package.module.Class.method 형태의 문자열로 받는다.
    import된 경로를 잘 확인한 후 적자!
  2. test_example_view(self, mock_example)
    patch된 mock 객체를 테스트 메서드의 매개변수로 받아 사용한다.
    한 테스트 함수에 여러 @patch 를 사용할 경우, 매개변수 순서에 유의하자.
    제일 위에 있는 @patch 의 mock 객체를 매개변수중 가장 마지막으로 받는다. (순서 거꾸로)
  3. mock_example.return_value = { 'key1': value1, 'key2': value2 }
    리턴될 값을 지정하는 것이다.
    테스트가 돌던 중 patch될 메서드로 지정한 example_api.get() 를 만났을때, 실제 example api를 호출하는 것이 아니라, return_value 로 지정된 가짜 데이터를 리턴한다.
  1. mock_example.side_effect = Exception('Error!')
    테스트가 돌던 중 patch될 메서드로 지정한 example_api.get() 을 만났을 때, 어떤 값을 return 하는게 아니라, Exception('Error!') 익셉션을 raise 한다.
    예외처리가 잘 되었는지 테스트할 때 유용하다.

오늘의 결론!

Mock객체의 속성을 잘 활용하자!

  • 반환 값 변경만 하면 될 때 : return_value 지정
  • 구현부 수정이 필요할 때 : side_effect에 커스텀 함수 지정
  • 예외처리 테스트가 필요할 때 : side_effect 에 Exception 클래스 지정

[Django] 필요한 필드만 골라서 조회하기 values(), values_list(), only()

Apr 14, 2021 in Django

우리가 장고 모델에서 필요한 필드만 골라서 조회를 하려면 크게 두 가지 방법이 있다.

  1. values() 와 values_list() 메소드 사용하기
  2. only() 메소드 사용하기

이 글에서는 총 세 개의 메소드에 대해 다룰 것이지만 크게 두 가지 방법이 있다고 한 이유는, values()와 values_list()는 바로 값을 가진 쿼리셋을 반환하지만, only()는 모델 객체를 가진 쿼리셋을 반환하기 때문이다.

쿼리셋의 결과를 알아두고, 필요에 맞게 적절히 사용하자.

1. values()

  • 딕셔너리를 가진 쿼리셋을 반환한다. : {'필드명' : '값', ...}
  • 필드를 지정하지 않으면 모든 필드를 반환하고, 필드를 지정하면 지정한 필드만 반환한다.

2. values_list()

  • 튜플을 가진 쿼리셋을 반환한다. : ('값', ...)
  • 필드를 지정하지 않으면 모든 필드를 반환하고, 필드를 지정하면 지정한 필드만 반환한다.

values_list() 의 flat=True 옵션

  • 필드 외에, flat 파라미터를 True로 지정하면, 튜플이 아닌 단일값을 반환한다. : '값'
  • 이 경우에는 1개의 필드만 지정할 수 있다.

3. only()

  • 모델 객체를 가진 쿼리셋을 반환한다. : <User: User object>
  • 지정된 필드 외에 id 를 항상 함께 조회한다.

주의점 : first() 처럼 한 row의 전 필드를 가져오는 특징을 가진 메서드와 함께 사용하면, only를 사용하더라도 전체조회를 하게된다.

위 예제처럼 호출시에 반환되는 User 객체에는, ‘name’, ‘birth’ 필드 뿐만아니라 다른 값도 함께 반환된다.
(실제로 only()에 해당하는 쿼리 한번, first()에 해당하는 쿼리 한번 해서 총 2번의 쿼리를 생성한다.)
따라서 only()와 first()를 함께 쓰는 것은 잘못된 방식이며, 이는 아래와 같은 방식으로 구현할 수 있다.
