[Database] 인덱스 튜닝과 설계

Jul 13, 2022 in Database

인덱스 튜닝과 설계에 대해 알아보자.

1. 인덱스 기본 구조

가장 일반적으로 사용되는 BTree 인덱스 구조부터 살펴보자!


  • 루트와 브랜치 블록 : 하위 노드들의 데이터 값 범위를 나타내는 키값 & 그 키값에 해당하는 블록을 찾는데 필요한 주소정보 를 가짐
  • 리프 블록 : 인덱스 키값 & 그 키값에 해당하는 테이블 레코드를 찾아가는데 필요한 주소정보(ROWID) 를 가짐
    • 키 값이 같을 때는 ROWID 순으로 정렬된다는 사실을 기억하자!
    • 리프 블록은 항상 인덱스 키값 순으로 정렬되어 있기 때문에, 범위스캔(Range Scan), 정방향(Ascending), 역방향(Descending) 스캔이 둘 다 가능하도록 Double Linked list 구조로 연결되어있음

2. 다양한 인덱스 스캔 방식

  1. Index Range Scan


    • 루트 ~ 리프 블록까지 수직 탐색 이후, 리프블록을 필요한 범위만 스캔하는 방식
    • 인덱스의 가장 일반적이고 정상적인 형태의 액세스 방식
    • 하지만 실행계획상에 Index Range Scan 이 나타난다고 해서 항상 빠른 속도를 보장하는 것은 아님!
      • 스캔 범위를 얼마만큼 줄일수 있느냐, 테이블로의 액세스 횟수를 얼만큼 줄일수 있느냐가 관건이다.
      • Index Range Scan 과정을 거쳐 생성된 결과집합은 인덱스 컬럼 순으로 정렬된 상태가 되기 때문에 이러한 특징을 잘 이용하면 sort order by 연산을 생략하거나 min/max 연산을 빠르게 할수있다.

  2. Index Full Scan


    • 수직 탐색 없이, 리프블록을 처음부터 끝까지 수평적으로 탐색하는 방식
    • 대개는 데이터 검색을 위한 최적의 인덱스가 없을 때 차선으로 선택됨
    • 인덱스 선두 컬럼이 조건절에 없으면 옵티마이저는 우선적으로 Table Full Scan 을 고려하지만, 대용량 테이블이어서 Table Full Scan 의 부담이 크다면 옵티마이저는 인덱스를 활용한 Index Full Scan 방식을 고려한다.
      • 하지만! 이러한 방식은 적절한 인덱스가 없어 Index Range Scan 의 차선책으로 선택된 것으로, 할수있다면 인덱스 구성을 조정해주는것이 좋다.

  3. Index Unique Scan


    • 수직적 탐색만으로 데이터를 찾는 스캔방식
    • Unique 인덱스를 ‘=’ 조건으로 탐색하는 경우에 작동한다.

  4. 그 외

    • Index Skip Scan
    • Index Fast Full Scan
    • Index Range Scan Descending
    • 등..

3. 인덱스 튜닝

인덱스를 정상적으로 사용하려면, 범위 스캔 시작지점을 찾기 위해 루트~리프 블록 까지의 수직적 탐색 과정을 거쳐야한다. 인덱스 선두 컬럼이 조건절에 사용되지 않으면 범위 스캔을 위한 시작점을 찾을수 없어 옵티마이저는 인덱스 Full 스캔하거나, 테이블 Full 스캔하는 방식을 선택한다.

인덱스 선두 컬럼이 조건절에 사용 되더라도 Range 스캔이 불가능하거나 , 아예 인덱스를 사용하지 못하는 경우가 어떤경우인지 살펴보자!

3-1. Index Range Scan 이 불가능하거나 아예 인덱스 사용이 불가능 한 경우

  1. 조건절에서 인덱스 컬럼을 가공했을 경우

  2. 부정형 비교를 사용했을 경우

    위 경우 정상적인 Index Range Scan 이 불가능하다. Index Full Scan 은 가능!

  3. 묵시적 형변환을 사용했을 경우
    사용자가 명시적으로 컬럼을 가공하지 않더라도, 조건절에서 비교되는 두 값의 데이터 타입이 다르면 내부적으로 형변환이 일어난다.

    • 문자열 → 숫자로 형변환됨
    • 문자형 → 날짜형으로 형변환됨
    • cdeptno가 인덱스 컬럼이고, 문자열 컬럼이다. 숫자형 20 으로 값 비교를 했을시 cdeptno 컬럼이 형변환이 되기 때문에 Full Scan 을 하게된다.
    • 데이터 타입간의 묵시적 형변환은 사용자가 코딩을 쉽게 하도록 도울 목적으로 DBMS 가 제공하는 기능인데, 위와같은 부작용을 피하려면!
      • 굳이 형변환 우선순위를 외울 필요없이 인덱스 컬럼 데이터 타입에 맞춰서 값비교를 하면 된다!

3-2. 테이블 Random 액세스 최소화

emp 테이블에 [deptno + job] 순으로 구성된 인덱스 하나만 있는 상태에서 아래 쿼리를 수행하려한다.

  • 위 조건을 만족하는 사원은 단 한명 뿐인데, 이를 찾기위해 테이블 액세스는 6번 발생한다.
  • 인덱스 구성을 [deptno + sal] 순으로 바꾸면 좋겠지만 실 운형 환경에서는 기존인덱스를 사용하는 쿼리가 있거나 등의 이유로 인덱스 구성을 함부로 바꾸기가 쉽지않다.
  • 이때 기존 인덱스에 sal 컬럼을 추가하는 것만으로 큰 효과를 거둘 수 있다. 인덱스 스캔량은 줄지 않지만 테이블 Random 액세스 횟수를 줄여주기 때문이다.

3-3. 인덱스 스캔범위 최소화

인덱스 선행 컬럼이 범위조건일 때는 상당한 비효율을 발생시킨다.

  • 인덱스 구성 컬럼이 조건절에서 모두 ‘=’ 조건으로 비교되면, 리프 블록을 스캔하면서 읽은 레코드는 모두 테이블 엑세스로 이어지기 때문에 읽고서 버리는 레코드가 하나도 없다.
  • 인덱스 컬럼중 일부가 ‘=’ 조건이 아니거나 조건절에서 생략되더라도 그것이 뒤쪽 컬럼일 때는 비효율이 없다.

아래와 같은 SQL 일 경우 인덱스 순서에 따른 효율성을 알아보자.

인덱스가 [아파트시세코드 + 평형 + 평형타입 + 인터넷 매물] 순으로 구성된 상황에서는
인터넷 매물이 between 조건이지만 선행컬럼들이 모두 ‘=’ 조건이기 때문에 전혀 비효율없이 조건을 만족하는 2건을 빠르게 찾는다.

인덱스 구성이 [인터넷메물 + 아파트 시세코드 + 평형 + 평형타입] 순으로 구성되어있다면
인덱스 스캔범위가 넓어진다. 따라서 조건을 만족하지 않는 레코드 까지 스캔하고서 버려야하는 비효율이 생긴다.

4. 인덱스 설계

결합 인덱스 구성을 위한 기본 공식

  1. 인덱스 선두 컬럼이 조건절에 반드시 사용되어야만 한다. 아니면 적어도 자주 사용되는 컬럼들을 선정하자.
  2. ‘=’ 조건으로 자주 조회되는 컬럼을 앞쪽에 두자.
  3. Sort Operation 을 생략하도록 하기 위해 컬럼을 추가하자. 인덱스는 항상 정렬상태를 유지하므로 order by, group by 를 위한 Sort 연산을 생략할 수 있도록 해준다.
    • 인덱스를 통해 Sort 연산을 대체하려면, 인덱스 컬럼 구성과 같은 순서로 누락없이 (뒤쪽 컬럼이 누락되는것은 상관없음) order by 절에 기술해야한다.
    • 단, 조건절에서 ‘=’연산자로 비교된 컬럼이 있다면 그 컬럼은 order by 절에서 누락되거나 다른 인덱스와 다른 순서로 기술하더라도 상관 없다. 이 규칙은 group by 에도 똑같이 적용된다.
  4. 선택도가 충분히 낮은지를 꼭 판단하자.
    1. 인덱스를 스캔하면서 테이블을 액세스 하는 양이 일정 수준을 넘는 순간 Table Full Scan 보다 오히려 늦어지기 대문에 선택도가 높은 인덱스는 생성해봤자 효용가치가 없다.
      선택도란 ? 전체 대상 레코드 중에서 특정 조건에 의해 선택될 것이라 예상되는 레코드 비율
  5. 추가적인 고려사항?
    1. 인덱스 스캔외에도 고려해야할 사항이 훨씬 많다.
      1. 쿼리 수행 빈도
      2. 업무상 중요도
      3. 클러스터링 팩터
      4. 데이터양
      5. DML 부하 (기존 인덱스 개수, 초당 DML 발생량, 자주 갱신되는 컬럼 포함 여부 등)
      6. 저장 공간
      7. 인덱스 관리 비용 등

인덱스 설계는 공식이 아닌 전략과 선택의 문제이다.
시스템 전체적인 관점에서 대안 전략들을 수립하고 그중 최적을 선택할 수 있는 고도의 기술력과 경험이 요구되기 때문에 어렵다. 개별 쿼리 성능을 높일 뿐만아니라, 생성되는 인덱스 개수를 최소화 함으로써 DML 부하를 줄이는 것이 중요한 목표이어야 한다.

오늘의 결론!

옵티마이저는 DBMS의 똑똑한 핵심 엔진이지만! 어디까지나 사용자가 적절한 옵티마이징 팩터(효과적으로 구성된 인덱스, 클러스터링, 파티셔닝 등) 를 제공하지 않는다면 결코 좋은 실행계획을 수립할 수 없다는 것을 인지하자!

[Database] 정규화와 반정규화의 성능

May 11, 2022 in Database

성능 데이터 모델링과 정규화, 반정규화의 성능에 대해 알아보자.

1. 성능 데이터 모델링의 개요

1-1. 성능 데이터 모델링의 정의
  1. 데이터베이스에서 기본적으로 설계단계에서부터 성능을 고려하지 않고 설계를 하는것은 빠르게 지나갈 수 없는 길을 지나가는 차에게 빨리 와달라고 요청하는 것과 다름없다.
  2. 일반적으로 성능이라고 하면 데이터 조회의 성능을 의미하곤 한다.
  3. 데이터 입력/수정/삭제는 일시적이고 빈번하지 않지만 데이터 조회의 경우 반복적이고 빈번하며 여러건을 처리하는 경우가 많기 때문이다.
  4. 성능 데이터 모델링이란 데이터베이스 성능향상을 목적으로 설계단계의 데이터모델링 때부터 정규화,반정규화, 테이블통합, 테이블분할, 조인구조, PK, FK 등 여러가지 성능과 관련된 사항이 데이터 모델링에 반영될 수 있도록 하는 것이다.
1-2. 성능 데이터 모델링 수행 시점
  1. 프로젝트 수행 중에 있어서 사전에 할수록 비용이 들지 않는다. 분석/설계 단계에서 데이터 모델에 성능을 고려한 데이터 모델링을 수행할 경우 성능저하에 따른 재업무 비용을 최소화 할 수 있는 기회를 가지게 된다.

2. 정규화와 성능

2-1. 정규화를 통한 성능 향상 전략

  1. 정규화란 ?

    • 기본적으로 데이터에 대한 중복을 제거하여 (같은 데이터는 DB 내에 하나만 놓으려고 함) 데이터가 관심사별로 처리되도록 하는 경우가 많기 때문에 성능이 향상되는 특징을 가진다.
    • 데이터 모델링을 할때 기본적으로 정규화가 완벽하게 수행되어 있어야한다.
    • 다만! SQL 문장에서 조인이 많이 발생하여 이로인한 성능저하가 나타나기 때문에 사례별로 유의하여 반정규화를 적용하는 전략이 필요하다.

  2. 그러면 정규화를 수행하면 항상 성능이 저하되어 나타날까 ?

    • 정규화는 데이터의 중복속성을 제거하고 동일한 의미의 속성들이 하나의 테이블로 집약되므로 한 테이블의 데이터 용량이 최소화 되는 효과가 있다. 따라서 정규화된 테이블은 데이터 처리할 때 속도가 빨라질 수도 있고 느려질 수도 있는 특성이 있다.

    • 일반적으로 정규화가 잘 되어있으면 입력/수정/삭제의 성능이 향상되고, 반정규화를 많이하면 조회의 성능이 향상된다고 인식할 수 있으나!! 그런 고정관념을 탈피하자.

    • 정규화를 해야만 성능이 향상되는 경우가 아주 많기 때문이다.

      사례 보기
      1. 사례 1
      • ‘관서등록일자가 2020년 이후 관서를 모두 조회하라’ 일 경우
      • 왼쪽 테이블(비정규화)은 불필요하게 납부자번호 만큼 누적된 데이터를 읽어서 결과를 보여주지만, 오른쪽 테이블(정규화)은 관서수 만큼만 존재하는 데이터를 읽어 곧바로 결과를 보여준다(PK Unique Index 이용하여 조인 성능저하 미미함).
      1. 사례 2
      • 유형기능분류코드에 따라 데이터를 조회하는 경우가 많이 나타나 인덱스를 생성하려면
      • 왼쪽 테이블(비정규화) 의 경우 각각에 대해 인덱스를 생성해야 하므로 9개나 되는 인덱스를 추가해야하지만, 오른쪽 테이블(정규화) 의 경우 PK 인덱스(유형코드 + 기능분류코드+모델코드) 를 이용하기 때문에 추가로 생성하는 인덱스의 개수가 0개이다.
      • 한 테이블에 인덱스가 많아지면 조회성능은 향상되지만, 데이터 입력/수정/삭제 성능은 저하된다. 인덱스 수를 가급적 7~8개가 넘지 않도록 하는것이 좋다.

3. 반정규화와 성능

3-1. 반정규화를 통한 성능 향상 전략
  1. 반정규화를 통한 성능향상 전략
    1. 반정규화(역정규화) ?
      1. 비정규화와 다르다!!! 비정규화는 아예 정규화를 수행하지 않은 모델을 지칭한다.
      2. 반정규화(역정규화)는 정규화된 엔티티, 속성, 관계에 대해 시스템 성능향상과 단순화를 위해 중복, 통합, 분리 등을 수행하는 데이터 모델링 기법이다. → 이말은 일단 베이스는 정규화된 데이터모델이어야한다는 것
    2. 데이터의 무결성이 깨질 수 있는 위험을 무릅쓰고도 반정규화를 적용하는 이유!?
      1. 데이터를 조회할 때 디스크 I/O량이 많아서 성능이 저하되거나, 경로가 너무 멀어 조인으로 인한 성능저하가 예상되거나, 컬럼을 계산하여 읽을 때 성능이 저하될 것이 예상되는 경우 반정규화를 수행한다.
  2. 반정규화의 적용방법
    1. 반정규화를 적용할 때는 기본적으로 데이터 무결성이 깨질 가능성이 많이 있기 때문에 반드시 데이터 무결성을 보장할 수 있는 방법을 다~ 고려해본 이후에 최후의 수단으로 반정규화를 적용해야 한다.
    2. 정규화 ↔  반정규화 는 Trade-Off 관계이기 때문에, 정규화만을 강조하다보면 성능의 이슈가 발생될 수 있고 반정규화를 과도하게 적용하다 보면 데이터 무결성이 깨질 위험이 증가한다.
  3. 반정규화의 절차
    1. 반정규화의 대상을 조사한다.
      1. 자주 사용되는 테이블에 접근하는 프로세스의 수가 많고 항상 일정한 범위만을 조회하는 경우인지
      2. 테이블에 대량의 데이터가 있고 대량의 데이터 범위를 자주 처리하는 경우에 처리범위를 일정하게 줄이지 않으면 성능을 보장할수 없는 경우인지
      3. 통계성 프로세스에 의해 통계 정보를 필요로 할 때 인지
      4. 테이블에 지나치게 많은 조인이 걸려 데이터를 조회하는 작업이 기술적으로 어려운 경우인지
    2. 반정규화의 대상에 대해 다른 방법으로 처리할 수 있는지 검토한다.
      1. 지나치게 많은 조인이 걸려 데이터를 조회하는 작업이 기술적으로 어려울 경우 뷰(VIEW) 를 사용하면 이를 해결 할 수도 있다.
      2. 성능이 저하되는 경우 인덱스를 조정함으로써 성능을 향상시킬 수 있다.
      3. 어플리케이션 단에서 로직을 구사하는 방법을 변경하여 성능을 향상시킬 수 있다.
      4. 대량의 데이터는 파티셔닝 기법을 적용하여 성능을 향상 시킬수 있다. 데이터가 특정 기준(파티셔닝 키)에 의해 다르게 저장되고 파티셔닝 키에 따른 조회가 될 때 성능이 좋아진다.
  4. 반정규화를 적용한다.
    1. 테이블과 속성, 관계에 대해 중복으로 가져가는 방법만이 반정규화는 아니고, 테이블, 속성, 관계를 추가할 수도 있고 분할할 수도 있으며 제거하는 방법도 반정규화다!
    2. 반정규화의 기법
      1. 테이블 반정규화
        1. 테이블 병합
          1. 1:1 관계 병합 : 1:1 관계를 통합하여 성능향상
          2. 1:M 관계 병합 : 1:M 관계를 통합하여 성능향상
          3. 슈퍼/서브타입 병합 : 슈퍼/서브 관계를 통합하여 성능향상
        2. 테이블 분할
          1. 수직분할 : 컬럼단위로 테이블을 1:1 로 분리하여 성능향상
          2. 수평분할 : 로우단위로 테이블을 분리하여 성능향상
        3. 테이블 추가
          1. 중복테이블 추가 : 서버가 다른 경우 동일한 테이블 구조를 중복하여 원격조인을 제거하며 성능향상
          2. 통계테이블 추가 : SUM, AVG 등을 미리 수행하여 계산해둠으로써 조회시 성능향상
          3. 이력테이블 추가 : 이력테이블 중에서 마스터테이블에 존재하는 레코드를 중복하여 관리함으로써 최신데이터 조회시 성능향상하는 것도 반정규화 유형중 하나
          4. 부분테이블 추가 : 자주이용하는 집중화된 컬럼들이 있을 때 해당 컬럼들을 모아놓은 별도의 테이블을 생성함으로써 성능향상
      2. 컬럼 반정규화
        1. 중복컬럼 추가 : 조인을 감소시키기 위해 중복된 컬럼을 위치시킴
        2. 파생컬럼 추가 : 트랜잭션이 처리되는 시점에 계산에 의해 발생되는 성능저하를 예방하기 위해 미리 값을 계산하여 컬럼에 보관
        3. 이력테이블 컬럼 추가 : 대량의 이력데이터를 처리할 때 불특정날조회나 최근값 조회할때 나타날 수 있는 성능저하를 예방하기위해 기능성컬럼(최근값여부, 시작 및 종료일자 등)을 추가함
        4. PK에 의한 컬럼 추가 : 복합의미를 갖는 PK를 단일 속성으로 구성하였을 경우 발생됨. 단일 PK안에서 특정 값을 별도로 조회하는 경우 성능저하가 발생할 수 있음. 이미 PK 안에 데이터가 존재하지만 성능향상을 위해 일반속성으로 포함하는 방법.
        5. 응용시스템 오작동을 위한 컬럼 추가 : 사용자가 데이터처리를 하다가 잘못 처리하여 원래값으로 복구하기를 원하는 경우 이전 데이터를 임시적으로 중복하여 보관하는 기법 (컬럼으로 이것을 보관하는 법은 임시적인 기법이지만, 이것을 이력데이터 모델로 풀어낸다면 정상적인 데이터 모델 기법이 됨)
      3. 관계 반정규화
        1. 중복관계 추가 : 데이터를 처리하기 위한 여러 경로를 거쳐 조인이 가능하지만 이 때 발생할 수 있는 성능저하를 예방하기 위해 추가적인 관계를 맺는 방법
        2. 테이블과 컬럼의 반정규화는 데이터 무결성에 영향을 미치게 되지만 관계의 반정규화는 데이터 무결성을 깨뜨릴 위험을 갖지 않고서도 데이터처리의 성능을 향상시킬 수 있는 반정규화의 기법이 된다.
