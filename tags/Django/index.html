[Django] Authentication 인증부를 구현하기 위해 알아야 할 것들

Oct 13, 2021 in Django, Authenticate

Authentication은 요청을 보낼 때 인증된 사용자나 Token 인지를 식별하는 역할을 한다. DRF에서 제공하고있는 인증 클래스를 이용하면 인증부를 구현할 수 있는데, 가장 기본적인 클래스는 BaseAuthentication 이 있다.

BaseAuthentication은 사용자의 username과 password를 기반으로 한 인증이기 때문에 가장 기본적이며, 보통 테스트 환경에서 적합한 인증 방법이다. 상용 환경에서 BaseAuthentication를 단독으로 사용하려면 API를 https를 통해서만 사용할 수 있는지, 항상 username/password 를 함께 요청할 수 있는지에 대한 고려가 필요하다.

따라서 보통은, 간단하게 Token 기반의 인증을 할 수 있는 TokenAuthentication을 사용한다.
TokenAuthentication 은 BaseAuthentication을 상속받아 구현이 되어있으며, 이번에는 TokenAuthentication을 이용하여 인증부를 구현할때 알아야 할 것들에 대해서 작성해본다.

기본적인 Authentication 클래스 DRF 설정에 관련한 설명은 아래 DRF 공식문서로 대체한다.

  • Authentication 설정하기 : DRF 공식문서

1. Token Model

TokenAuthentication 인증 클래스를 사용하려면 토큰 정보를 저장할 테이블이 필요하다. Token Model은 직접 구성하여 사용할 수도 있고, DRF의 rest_framework.authtoken APP을 추가하여 사용할 수도 있다. (django migration을 제공한다.)

Token 의 주된 필드는 key 와 user 이다.

  • key : Token 의 PK 이며, 인증에 사용되는 토큰 키다.
  • user : 해당 key 를 발급 받을 user를 말한다.
    • settings 파일에서 AUTH_USER_MODEL 에 User Model 을 지정하여 사용한다.

2. .authenticate() 알아보기

Authentication 인증 클래스의 핵심 메서드이다.
만약 인증 로직을 커스텀 하고 싶다면 .authenticate() 를 Override 하여 사용한다. 해당 메서드는 커스텀이 되더라도 아래 몇가지 기본적인 틀을 갖추고 있어야한다.

  • 인증이 성공하면 (user, auth) 형태의 Tuple을 return 해야한다.

  • 인증에 시도했지만 실패하는 경우에는 AuthenticationFailed Exception 을 raise 하거나, None 을 return 해야한다.

  • 인증이 아예 시도되지 않은 경우라면, None을 반환한다.

    • 사용중인 다른 Authentication 인증 Class 로직을 탈 수 있어야 하기 때문이다. (Authentication은 다수개 지정이 가능하다.)

    3번째 항목에서 알 수 있겠지만, .authenticate() 에서는 인증을 시도하지 않은 경우에 대한 예외처리가 되지 않는다. 이는 인증(authenticate)의 영역이 아니라, 권한(permission)의 영역이기 때문이다. 이에 대한 예외처리를 하기 위해서는 rest_framework에서 제공하는 IsAuthenticated Permission 을 사용한다.

    IsAuthenticated Permission 은 인증된 유저가 존재하지 않을 경우 Exception을 낸다. Permission 이라 403 Forbidden 응답을 낼 것 같지만, rest_framework 내부에서 인증되지 않는 경우를 분기처리하여 401 UnAuthorized 응답을 내린다.

3. TokenAuthentication의 .authenticate()

TokenAuthentication 에서는 기본적으로 HTTP header 에 담겨온 AUTHORIZATION 키를 검증한다.
그리고 값으로는 공백이 붙은 ‘Token ‘ 키워드로 시작하는지를 검증한다.

[Django] Unit Test의 Mocking! @patch

May 12, 2021 in Django

Mocking ?

Mock 은 ‘거짓된, 가짜의’ 라는 사전적 의미를 가진 것 처럼, 테스트코드를 작성할 때 가짜 값을 사용할 수 있게 하는 객체이다. 이 Mock 객체를 이용하여, 단위 테스트 시에 외부에 의존하는 부분을 임의의 가짜로 대체하여 테스트 하는 상황이 자주 발생하는데, 이것을 ‘Mocking’ 이라고 한다.

Mocking 방법은 2가지가 있다.

  1. 직접 Mock 객체를 생성하여 사용
  2. unittest.mock 모듈의 @patch 데코레이터를 사용

이 글은 Mocking시 주로 사용되는 Mock객체의 속성에 대해 먼저 알아보고, @patch 데코레이터를 사용하는 방법에 대해 작성한다.

1. Mock

@patch 에서 사용되는 Mock 객체의 주요 속성을 알아보자.

  • return_value: 반환되는 값을 설정한다.

  • side_effect : 설정하는 값에 따라 동작이 다르다.

    • Exception 클래스를 지정하면 에러를 발생한다.
    • 함수를 지정하면 해당 함수를 호출한다.
    • Iterable(ex.list)을 지정하면 mock 호출 순서대로 해당 값을 반환한다.

2. @patch

unittest.mock 모듈의 @patch 데코레이터를 이용하면, 지정한 특정 메서드를 Mocking하여 해당 메서드가 호출될 때 mock 객체로 patch 한다. patch된 mock 객체는 테스트가 끝나면 원래대로 복원된다.

  1. @patch('example_api.client.ExampleAPI.get')
    Mocking 이 필요한 메서드를 지정하는 부분이다.
    지정할 메서드의 path는 package.module.Class.method 형태의 문자열로 받는다.
    import된 경로를 잘 확인한 후 적자!
  2. test_example_view(self, mock_example)
    patch된 mock 객체를 테스트 메서드의 매개변수로 받아 사용한다.
    한 테스트 함수에 여러 @patch 를 사용할 경우, 매개변수 순서에 유의하자.
    제일 위에 있는 @patch 의 mock 객체를 매개변수중 가장 마지막으로 받는다. (순서 거꾸로)
  3. mock_example.return_value = { 'key1': value1, 'key2': value2 }
    리턴될 값을 지정하는 것이다.
    테스트가 돌던 중 patch될 메서드로 지정한 example_api.get() 를 만났을때, 실제 example api를 호출하는 것이 아니라, return_value 로 지정된 가짜 데이터를 리턴한다.
  1. mock_example.side_effect = Exception('Error!')
    테스트가 돌던 중 patch될 메서드로 지정한 example_api.get() 을 만났을 때, 어떤 값을 return 하는게 아니라, Exception('Error!') 익셉션을 raise 한다.
    예외처리가 잘 되었는지 테스트할 때 유용하다.

오늘의 결론!

Mock객체의 속성을 잘 활용하자!

  • 반환 값 변경만 하면 될 때 : return_value 지정
  • 구현부 수정이 필요할 때 : side_effect에 커스텀 함수 지정
  • 예외처리 테스트가 필요할 때 : side_effect 에 Exception 클래스 지정

Reference
https://python.flowdas.com/library/unittest.mock.html#unittest.mock.Mock.return_value
https://python.flowdas.com/library/unittest.mock.html#unittest.mock.Mock.side_effect

[Django] 필요한 필드만 골라서 조회하기 values(), values_list(), only()

Apr 14, 2021 in Django

우리가 장고 모델에서 필요한 필드만 골라서 조회를 하려면 크게 두 가지 방법이 있다.

  1. values() 와 values_list() 메소드 사용하기
  2. only() 메소드 사용하기

이 글에서는 총 세 개의 메소드에 대해 다룰 것이지만 크게 두 가지 방법이 있다고 한 이유는, values()와 values_list()는 바로 값을 가진 쿼리셋을 반환하지만, only()는 모델 객체를 가진 쿼리셋을 반환하기 때문이다.

쿼리셋의 결과를 알아두고, 필요에 맞게 적절히 사용하자.

1. values()

  • 딕셔너리를 가진 쿼리셋을 반환한다. : {'필드명' : '값', ...}
  • 필드를 지정하지 않으면 모든 필드를 반환하고, 필드를 지정하면 지정한 필드만 반환한다.

2. values_list()

  • 튜플을 가진 쿼리셋을 반환한다. : ('값', ...)
  • 필드를 지정하지 않으면 모든 필드를 반환하고, 필드를 지정하면 지정한 필드만 반환한다.

values_list() 의 flat=True 옵션

  • 필드 외에, flat 파라미터를 True로 지정하면, 튜플이 아닌 단일값을 반환한다. : '값'
  • 이 경우에는 1개의 필드만 지정할 수 있다.

3. only()

  • 모델 객체를 가진 쿼리셋을 반환한다. : <User: User object>
  • 지정된 필드 외에 id 를 항상 함께 조회한다.

주의점 : first() 처럼 한 row의 전 필드를 가져오는 특징을 가진 메서드와 함께 사용하면, only를 사용하더라도 전체조회를 하게된다.

위 예제처럼 호출시에 반환되는 User 객체에는, ‘name’, ‘birth’ 필드 뿐만아니라 다른 값도 함께 반환된다.
(실제로 only()에 해당하는 쿼리 한번, first()에 해당하는 쿼리 한번 해서 총 2번의 쿼리를 생성한다.)
따라서 only()와 first()를 함께 쓰는 것은 잘못된 방식이며, 이는 아래와 같은 방식으로 구현할 수 있다.

오늘의 결론!

  • key, value 형태의 딕셔너리 자료형이 필요한 경우에는 values()
  • value 를 가진 튜플 또는 단일 값이 필요한 경우에는 values_list()
  • 장고 모델이 필요한 경우에는 only()
